# 05 вҖ” File Permissions in Depth

## ELI5 вҖ” The Simple Analogy

Imagine a **self-storage facility**.

- Every **storage unit** (a file) has an owner, and belongs to a **rental tier** (a group). Everyone else on Earth is a stranger.
- The facility manager posts a little card on every unit with **three rows**: what the **owner** may do, what anyone in the **same tier** may do, and what a **stranger** may do. Three rows, three columns: *look inside* (r), *change what's inside* (w), *use the machinery in there* (x).
- Now the crucial part everyone gets wrong. A **corridor** (a directory) also has one of these cards, but the three letters mean **completely different things** for a corridor:
  - **r** = you may read the *list of unit numbers* posted at the corridor entrance.
  - **w** = you may **add or remove units from that list** вҖ” that is, you may *delete a unit* even if you have zero rights to what's inside it. The list is yours to edit, not the contents.
  - **x** = you may **walk down the corridor**. Without it, you can't reach anything past it вҖ” even if you know the exact unit number and hold its key.
- **`w` on the corridor is the power to delete a unit whose door you cannot even open.** This is not a bug. This is the design.
- The **sticky bit** is the manager's fix for the shared public corridor (`/tmp`): "anyone may drop off a unit here, but you may only remove *your own*."
- **setuid** is a special unit that comes with a **borrowed manager's keycard**: when *you* operate the machinery inside, it runs with the *manager's* authority, not yours. Immensely useful (`passwd` needs it). Immensely dangerous (leave one lying around and anyone becomes the manager).

## Where This Lives in the Linux Stack

```
Hardware

  в””в”Җв”Җ KERNEL
       в”Ӯ
       в”ңв”Җв”Җ The permission check itself в—Җв—Җв—Җ THIS TOPIC
       в”Ӯ     Every open(), every execve(), every unlink() passes through
       в”Ӯ     inode_permission() вҶ’ generic_permission() in fs/namei.c.
       в”Ӯ     It compares the process's (euid, egid, supplementary groups)
       в”Ӯ     against the inode's (i_uid, i_gid, i_mode). Returns 0 or -EACCES.
       в”Ӯ     вҡ  THE FILESYSTEM ENFORCES NOTHING. THE KERNEL DOES.
       в”Ӯ

            в”Ӯ

```

**Why it lives in the kernel:** if permission checks lived in userspace, any program could simply skip them. The check happens *inside* the syscall, after you've crossed into Ring 0, where your process cannot reach. Your Node.js `EACCES` was not generated by Node вҖ” it is `errno 13`, handed back by the kernel's `generic_permission()`.

## What Is This?

Every inode carries a **16-bit mode field**. Four bits say *what kind of file* this is; three bits are the special bits (setuid, setgid, sticky); the remaining **nine bits** are the famous `rwxrwxrwx` вҖ” three permissions Г— three identity classes (**user**, **group**, **other**).

When any process calls `open()`, `execve()`, or `unlink()`, the kernel picks **exactly one** of those three classes based on who you are, and checks **only that class**.

---

## Why Does This Matter for a Backend Developer?

| If you don't understand this... | This will happen... |
|---|---|
| `x` on a directory means *traverse*, not *execute* | You'll `chmod 644` a directory to "make it safe" and lock your entire app out of `/var/www` with `EACCES` вҖ” including root's own daemons |
| Deletion is controlled by the **parent directory** | You'll `chmod 400` a file to "protect" it, and it'll get deleted anyway |
| `umask` | Your deploy writes uploads that nobody else can read, or worse, that everybody can write |
| setgid on directories | Your shared deploy dir will grow files owned by the wrong group, and the CI user won't be able to overwrite yesterday's build |
| Why `chmod -R 777` is catastrophic | You'll "fix" a permission bug by opening the whole app to every process on the box вҖ” then SSH will stop working, because `sshd` refuses world-writable key files |
| The kernel checks **only one** class | You'll be baffled when the *owner* is denied access to a file the *group* can read |
| `namei -l` | You'll spend an hour on an `EACCES` caused by a missing `+x` **four directories up the path** |

### Octal derivation вҖ” do the math once and never guess again

Each triplet is a 3-bit number: `r`=4, `w`=2, `x`=1. Add them.

```
  rwx = 4+2+1 = 7      r-x = 4+0+1 = 5      r-- = 4 = 4      --- = 0
  rw- = 4+2   = 6      r-x                  -rw-r--r-- = 110 100 100 = 644
```

| Octal | Bits | Symbolic | Typical use |
|---|---|---|---|
| `777` | 111 111 111 | `rwxrwxrwx` | **Never.** See Mistake 3. |
| `755` | 111 101 101 | `rwxr-xr-x` | Directories; executables/scripts |
| `750` | 111 101 000 | `rwxr-x---` | A dir only the owner + one group may enter |
| `700` | 111 000 000 | `rwx------` | `~/.ssh`, a private app dir |
| `664` | 110 110 100 | `rw-rw-r--` | Shared-write file (use with setgid dirs) |
| `644` | 110 100 100 | `rw-r--r--` | Normal file: app source, HTML, JSON |
| `640` | 110 100 000 | `rw-r-----` | Log a group must read, world must not |
| `600` | 110 000 000 | `rw-------` | **`.env`, private keys, `authorized_keys`** |
| `400` | 100 000 000 | `r--------` | Read-only secret |

## How It Works вҖ” Step by Step

### The kernel's permission check вҖ” the part nobody tells you

```
Process (euid=1001 "deploy", egid=1001, groups=[1001,33 "www-data"])
opens /var/www/app/.env  (inode: uid=0 root, gid=33 www-data, mode=0640)

KERNEL вҖ” generic_permission():

  Step 0. Am I root (or do I hold CAP_DAC_OVERRIDE)?
          вҶ’ If yes: ALLOW immediately. (One exception: for execve(), root still
            needs at LEAST ONE x bit set somewhere in the mode.)
          вҶ’ Not root here. Continue.

  Step 1. Does process.euid == inode.i_uid?
          1001 == 0 ?  NO.

  Step 2. Does process.egid == inode.i_gid, OR is inode.i_gid in my supplementary groups?
          33 is in [1001, 33] вҶ’ YES.
          вҮ’ USE THE GROUP TRIPLET.  в—Җв—Җв—Җ CLASS CHOSEN. The check STOPS here.

  Step 3. Group triplet of 0640 is  r--  (4).
          I asked for O_RDONLY вҶ’ need r вҶ’ granted.
          If I had asked for O_WRONLY вҶ’ need w вҶ’ GROUP HAS NO w вҶ’ return -EACCES (errno 13).

  вҡ  THE THREE CLASSES ARE NOT ADDITIVE. The kernel picks ONE and checks ONLY it.
```

**The consequence that catches everyone:**

```bash
# mode 0466 = user: r--   group: rw-   other: rw-
touch weird.txt && chmod 0466 weird.txt && ls -l weird.txt
# -r--rw-rw- 1 deploy deploy 0 Jul 12 16:02 weird.txt

echo hi > weird.txt          # you are the OWNER
# bash: weird.txt: Permission denied
```
You own the file. Group and Other both have `w`. **You are still denied** вҖ” because `euid == i_uid` matched at Step 1, so the kernel used the **user** triplet (`r--`) and never looked at the others. Owning a file can give you *less* access than a stranger.

### `rwx` on a FILE vs `rwx` on a DIRECTORY

This is the single most important table in this document.

| Bit | On a **file** | On a **directory** |
|---|---|---|
| **r** (4) | Read the file's **contents** | List the **names** inside (`ls`). But **not enough to `stat` them** вҖ” you get names and nothing else, because reading their inodes needs `x`. |
| **w** (2) | Modify the file's **contents** | **Create, delete, and rename entries** in this directory. Applies to ANY file inside, **regardless of that file's own permissions or owner.** Requires `x` as well to be usable. |
| **x** (1) | **Execute** it as a program (needs a valid shebang or ELF header) | **Traverse / search** вҖ” resolve a name through this directory to reach what's inside. Also called the "search bit." Needed for every single component of a path. |

**The two classic traps:**

**Trap A вҖ” you can enter a directory you cannot list.**
```bash
mkdir vault && echo "secret" > vault/key.txt && chmod 711 vault    # rwx--x--x
# As another user:
ls  /home/deploy/vault           # cannot open directory: Permission denied   вҶҗ no r
cat /home/deploy/vault/key.txt   # secret                                     вҶҗ x lets you TRAVERSE
```
`x` without `r` = "you may pass through **if you already know the name.**" Exactly how `/home` is often configured вҖ” and how "security by obscurity" directories work, and fail.

**Trap B вҖ” you can DELETE a file you have no write permission on.**
```bash
sudo touch /home/deploy/important.conf && sudo chmod 444 /home/deploy/important.conf  # root:root
# As deploy (NOT root):
echo x >> important.conf         # Permission denied   вҶҗ cannot WRITE it. Good.
rm -f important.conf             # ...GONE anyway.
```
**Why:** `unlink()` removes a **directory entry** (topic 04). The kernel checks `w`+`x` on the **parent directory** вҖ” which `deploy` owns вҖ” and **never looks at the file's own mode.** (`rm`'s "write-protected?" prompt is a GNU courtesy, not a kernel check; `rm -f` and Node's `unlink()` skip it.) The fix for the shared case is the **sticky bit** вҖ” keep reading.

### Path resolution вҖ” where `EACCES` actually comes from

Opening `/var/www/app/uploads/x.png` requires:

```
  /                     вҶ’ x   (traverse)
  /var                  вҶ’ x   (traverse)
  /var/www              вҶ’ x   (traverse)
  /var/www/app          вҶ’ x   (traverse)
  /var/www/app/uploads  вҶ’ x   (traverse)   + w if creating a NEW file in it
  x.png                 вҶ’ r or w on the FILE itself

  ONE missing x anywhere in that chain вҶ’ EACCES on the whole open().
  The error message names the FULL PATH, giving you no clue WHICH component failed.
  This is what `namei -l` exists to solve.
```

## umask вҖ” Why Your New Files Are Never Executable

The kernel never invents a mode. **The creating program supplies one**, and the kernel then subtracts the process's `umask`.

```
  open("app.log", O_CREAT, 0666)   вҶҗ what nearly EVERY program requests for a FILE
  mkdir("uploads", 0777)           вҶҗ what nearly EVERY program requests for a DIR

  final_mode = requested_mode & ~umask        With the default umask 022:

  FILE:  0666 (rw-rw-rw-) & ~022  вҶ’  0644  rw-r--r--   вң“
  DIR:   0777 (rwxrwxrwx) & ~022  вҶ’  0755  rwxr-xr-x   вң“
```

**Why is a new file never executable?** Because programs request **0666, not 0777**, for regular files вҖ” the `x` bits were never asked for, so umask isn't even involved. The kernel cannot grant a bit the caller didn't request. That is why a freshly-written script *always* needs `chmod +x`.

```bash
umask            # 0022  вҶҗ the default on Ubuntu/Debian for non-system users
umask 077 && touch secret.txt && mkdir secretdir && ls -ld secret.txt secretdir
# -rw------- deploy deploy secret.txt  вҶ’ 600     drwx------ deploy deploy secretdir вҶ’ 700
umask 022        # put it back
```

| umask | New files | New dirs | Use it for |
|---|---|---|---|
| `022` | 644 | 755 | Default. World-readable. |
| `002` | 664 | 775 | Shared group development (pairs with setgid dirs) |
| `027` | 640 | 750 | **Hardened servers** вҖ” group reads, world sees nothing |
| `077` | 600 | 700 | Secrets, single-user daemons |

**Trap:** umask is a **per-process** attribute, inherited across `fork()`/`execve()`. Your shell's umask (022, from `/etc/profile`) is **not** a systemd service's. Set it per-service:

```ini
# /etc/systemd/system/myapi.service
[Service]
User=nodeapp
UMask=0027          # вҶҗ files this daemon creates: 640, dirs: 750
```

## The Special Bits вҖ” setuid, setgid, sticky

Three extra bits sit above the nine `rwx` bits, forming a **fourth octal digit** at the front.

```
  chmod 4755 file      chmod 2775 dir       chmod 1777 dir
        в”Ӯ                    в”Ӯ                    в”Ӯ
        в””в”Җв”Җ setuid (4)       в””в”Җв”Җ setgid (2)      в””в”Җв”Җ sticky (1)
```

### setuid (4000) вҖ” "run as the file's OWNER"

```bash
ls -l /usr/bin/passwd
# -rwsr-xr-x 1 root root 68208 Feb 6 12:34 /usr/bin/passwd
#    ^ 's' in the USER-x slot = "x AND setuid". (Uppercase 'S' = setuid WITHOUT x вҖ” a bug.)
```

**Why it must exist:** `passwd` must write `/etc/shadow` (`-rw-r----- root:shadow`), to which a normal user has zero access. When you (uid 1001) `execve("/usr/bin/passwd")`, the kernel sees the setuid bit and sets the process's **effective UID to the file's owner (root)** вҖ” so for its lifetime it *is* root, edits only your line, then drops privilege. This is also the **#1 local privilege-escalation vector**: a setuid-root binary with an overflow or a `$PATH`-respecting `system()` call hands an attacker a root shell.

**Linux IGNORES setuid on shell scripts** вҖ” deliberately:
```bash
printf '#!/bin/bash\nid -u\n' | sudo tee /tmp/danger.sh >/dev/null
sudo chown root /tmp/danger.sh && sudo chmod 4755 /tmp/danger.sh
/tmp/danger.sh                   # 1001  вҶҗ the bit IS set, and was IGNORED. Still you.
```
The shebang means the kernel really execs `/bin/bash /tmp/danger.sh`, opening a TOCTOU race where an attacker swaps the file between the check and the open. So setuid is honoured **only on real ELF binaries**. Need a privileged script? Use `sudo` with a tight `sudoers` rule (topic 32) вҖ” never a setuid wrapper.

```bash
# AUDIT вҖ” anything here NOT from a distro package is an incident:
sudo find / -xdev -type f -perm -4000 2>/dev/null
# /usr/bin/passwd   /usr/bin/sudo   /usr/bin/mount   /usr/bin/su ...
```

### setgid (2000) вҖ” two totally different behaviours

**On a FILE:** the process runs with the file's **group** as its effective GID (rare вҖ” `wall`, `write` are setgid `tty`).

**On a DIRECTORY (the one you'll use):** new files and subdirectories inside **inherit the directory's group** instead of the creator's primary group, and subdirs inherit the setgid bit too, so it propagates down.

```bash
sudo mkdir -p /var/www/shared && sudo chown root:www-data /var/www/shared
sudo chmod 2775 /var/www/shared            # 2 = setgid, 775 = rwxrwxr-x
# As user `deploy` (primary group `deploy`):
touch /var/www/shared/build.js && ls -l /var/www/shared/build.js
# -rw-rw-r-- 1 deploy www-data 0 ... build.js    вҶҗ group is www-data вҖ” INHERITED, not deploy!
```

**Why it's critical for deploys:** without it, `deploy` writes files owned `deploy:deploy` that the nginx `www-data` user can't read. With `2775` + a `002` umask every artifact lands `deploy:www-data` mode `664` вҖ” group-writable, nginx-readable, forever, with no `chown -R` in your deploy script.

### Sticky bit (1000) вҖ” "you may only delete your OWN files here"

Trap B means `w` on a directory lets you delete *anything* in it вҖ” which would make a world-writable `/tmp` unusable (any user could nuke any other's temp files or PID/socket files).

```bash
stat -c '%a %A %n' /tmp
# 1777 drwxrwxrwt /tmp
#           ^ 't' = sticky (lowercase = x also set; 'T' = sticky without x). Mode 1777.
```

With the sticky bit set, `unlink()`/`rename()` in that directory additionally requires **you own the file, OR you own the directory, OR you are root** вҖ” `w` alone is no longer enough.

```bash
sudo -u nobody touch /tmp/not-mine
rm /tmp/not-mine
# rm: cannot remove '/tmp/not-mine': Operation not permitted
#     вҶҗ EPERM, not EACCES. A DIFFERENT errno вҖ” it's the sticky check, not the rwx check.
```
Use `1777` on any drop-box directory (`/tmp`, `/var/tmp`, a shared upload spool).

### The special-bit cheat sheet

| Bit | Octal | On a file | On a directory | `ls -l` shows |
|---|---|---|---|---|
| setuid | `4000` | Run as file's **owner** | *(nothing on Linux)* | `s` (or `S` if no x) in the **user**-x slot |
| setgid | `2000` | Run as file's **group** | **New entries inherit the dir's group**; subdirs inherit the bit | `s` (or `S`) in the **group**-x slot |
| sticky | `1000` | *(nothing on Linux)* | **Only the file's owner may delete/rename it** | `t` (or `T` if no x) in the **other**-x slot |

вҡ  **`chmod 755 f` silently clears all three special bits** (a 3-digit octal writes all 12 low bits, with the top digit implied `0`). If you mean to keep them, use 4 digits (`chmod 4755`) or symbolic mode (`chmod u+x`). A `chmod -R 755 /usr/bin` would de-fang `sudo` and `passwd` and break your system.

## Example 1 вҖ” Basic

```bash
mkdir -p /tmp/perm-lab && cd /tmp/perm-lab
printf '#!/bin/bash\necho "I ran as: $(id -un)"\n' > hello.sh
ls -l hello.sh
# -rw-r--r-- 1 deploy deploy 38 ... hello.sh
#  ^^^^^^^^^ 644 вҖ” `>` calls open(...,0666), umask 022 trims it. NO x ANYWHERE: never requested.

./hello.sh                   # bash: ./hello.sh: Permission denied  вҶҗ execve() failed, no x
chmod u+x hello.sh           # вҶ’ -rwxr--r-- (744)
./hello.sh                   # I ran as: deploy   вҶҗ now execve() succeeds

# The shebang: the kernel reads the FIRST TWO BYTES. If '#!', it runs the named interpreter.
# The x bit and the shebang are TWO SEPARATE requirements:
chmod 644 hello.sh && bash hello.sh
# I ran as: deploy   вҶҗ works! You executed BASH (has x); bash merely READ hello.sh (needs r,
#                      not x). That's why `bash script.sh` runs a non-executable script but
#                      `./script.sh` does not.

# --- Directory bits: r=list, x=traverse, and they're INDEPENDENT ---
mkdir dir && echo "contents" > dir/file.txt

chmod 400 dir                # r-- : NO x
ls dir                       # file.txt        вҶҗ can LIST the name...
cat dir/file.txt             # Permission denied  вҶҗ ...but can't TRAVERSE to the inode

chmod 100 dir                # --x : NO r
ls dir                       # cannot open directory  вҶҗ can't LIST...
cat dir/file.txt             # contents        вҶҗ ...but x lets you traverse IF you know the name

chmod 700 dir                # restore sanity
```

## Example 2 вҖ” Production Scenario

**11:40 PM.** Your Node API returns `500` on every avatar upload. The Slack channel is on fire. You SSH into the box.

```bash
# The app's own log, first.
sudo journalctl -u myapi -n 5 --no-pager
# Jul 12 23:38:04 prod-api node[8234]: Error: EACCES: permission denied, open '/var/www/app/uploads/a7f3.png'
# Jul 12 23:38:04 prod-api node[8234]:     at Object.openSync (node:fs:601:3)
# Jul 12 23:38:04 prod-api node[8234]:     at writeFileSync (node:fs:2249:35)
```

`EACCES` is **errno 13**. It came from the **kernel**, not from Node (topic 01). Something in the path denied the `open(..., O_WRONLY|O_CREAT)`.

Instinct says `ls -l /var/www/app/uploads`. **That's the mistake** вҖ” it only shows you the last component. The failure could be **any** directory in the chain.

```bash
id nodeapp                        # WHO is the app? (systemctl show myapi -p User confirms it)
# uid=998(nodeapp) gid=998(nodeapp) groups=998(nodeapp)   вҶҗ NOT in www-data. Remember this.

namei -l /var/www/app/uploads/a7f3.png    # THE command: walk the ENTIRE path
```
```
f: /var/www/app/uploads/a7f3.png
 drwxr-xr-x root    root     /
 drwxr-xr-x root    root     var
 drwxr-xr-x root    root     www
 drwxr-x--- deploy  www-data app        в—Җв—Җв—Җ FOUND IT вҖ” 'other' has NO bits
 drwxr-xr-x deploy  deploy   uploads        (uploads itself is fine!)
                             a7f3.png - No such file or directory
```

**Read it:** the process `nodeapp` is not owner `deploy`, and `id` proved it's not in group `www-data` either вҶ’ the kernel falls to the **other** class of `app` вҶ’ `---` вҶ’ **no `x`, cannot traverse.** It never even reaches `uploads`, whose bits are perfect. `ls -l uploads` would have shown nothing wrong and cost you 30 minutes. Root cause: last week's deploy ran `chmod 750 /var/www/app` to "harden" it.

```bash
# FIX вҖ” least privilege: add the service account to the group that owns the dir.
sudo usermod -aG www-data nodeapp
sudo systemctl restart myapi      # вҡ  group membership is read at process START вҖ”
                                  #   a running process keeps its old group list!
sudo -u nodeapp test -x /var/www/app && echo "traverse OK"   # ASK the kernel, don't guess
# traverse OK

# Make future uploads land as group www-data automatically (setgid), so nginx can serve them:
sudo chgrp www-data /var/www/app/uploads && sudo chmod 2775 /var/www/app/uploads
curl -s -o /dev/null -w '%{http_code}\n' -F 'file=@a.png' localhost:3000/upload    # 200
```

**The correct permission layout for a Node app вҖ” memorize this:**

```
/var/www/app/       755  deploy:www-data   drwxr-xr-x   dirs need x to traverse
в”ңв”Җв”Җ server.js       644  deploy:www-data   -rw-r--r--   source: read-only to the world
в”ңв”Җв”Җ .env            600  nodeapp:nodeapp   -rw-------   в—Җ SECRETS. OWNER-ONLY, owned by the
в”Ӯ                                                         RUNTIME user, not the deploy user.
в”ңв”Җв”Җ node_modules/   755  deploy:www-data   drwxr-xr-x
в”ңв”Җв”Җ uploads/       2775  deploy:www-data   drwxrwsr-x   в—Җ setgid: app WRITES here
в””в”Җв”Җ logs/app.log    640  nodeapp:adm       -rw-r-----   ops can read, world cannot
```

Fix an entire tree in one pass, without making every `.js` file executable:
```bash
sudo chown -R deploy:www-data /var/www/app
sudo chmod -R u=rwX,g=rX,o=rX /var/www/app     # capital X = x on DIRS only в—Җв—Җв—Җ
sudo chmod 600 /var/www/app/.env && sudo chown nodeapp:nodeapp /var/www/app/.env
sudo chmod 2775 /var/www/app/uploads
```

## Common Mistakes

### Mistake 1 вҖ” `chmod -R 755` (or `644`) on a directory tree

**Wrong:** `chmod -R 644 /var/www/app` ("make everything read-only, safe").
**Right:** `chmod -R u=rwX,g=rX,o=rX /var/www/app` (capital `X`), or the two-pass `find -type d -exec chmod 755` + `find -type f -exec chmod 644`.

**Root cause:** `-R 644` strips `x` from **directories too**. Without `x` a directory cannot be traversed вҖ” the kernel's `link_path_walk()` returns `-EACCES` there, and your app can open *nothing* beneath it; even `cd` fails. Root survives (it holds `CAP_DAC_OVERRIDE`), which is why "it works when I sudo" fools you into thinking it's a *file* problem.
**Diagnose:** `find /var/www -type d ! -perm -u+x`. **Prevention:** never use a raw numeric mode with `-R` on a mixed tree вҖ” capital `X` is the whole reason it exists.

---

### Mistake 2 вҖ” "I `chmod 400`'d it, so it can't be deleted"

**Wrong:** protecting a file by removing write permission *from the file*.
**Right:** protect the **directory**, or use the sticky bit / immutable flag.

**Root cause:** deletion is a write to the **parent directory's data block** (topic 04: `unlink()` removes a name), so the kernel checks `w`+`x` on the **parent** and never consults the file's own mode.
**Fix (three tiers):**
```bash
chmod a-w /var/www/app             # 1. nobody can modify the DIRECTORY listing
chmod +t  /var/spool/uploads       # 2. sticky: users delete only their OWN files
sudo chattr +i /etc/critical.conf  # 3. IMMUTABLE вҖ” even root can't unlink until `chattr -i`
```

---

### Mistake 3 вҖ” `chmod -R 777` "to make it work"

**Wrong:** `sudo chmod -R 777 /var/www` when you hit `EACCES`.
**Right:** find the *actual* denied component with `namei -l`, then grant the minimum bit.

**What 777 actually opens up:**

| It does this | Which means |
|---|---|
| **World-writable files** | Any process вҖ” a compromised cron job, a rogue container вҖ” can **rewrite `server.js`**. Your app runs the attacker's code on the next restart: persistent RCE. |
| **World-readable files** | `.env` (DB password, JWT secret, Stripe key) is readable by `nobody` and by any service that gets popped. |
| **Clears setuid/setgid/sticky** | Kills group inheritance on a setgid tree; strips the sticky bit off `/tmp` so any user can nuke any other's files. |
| **SSH breaks** | `sshd` runs `StrictModes yes` and **refuses** `~/.ssh/authorized_keys` if the file or home dir is group/world-writable. You lock yourself out and won't know why. |

**Diagnose:** `find /var/www -perm /o+w -ls`. **Fix:** restore a sane layout (Example 2), `chmod 600 .env`, and rotate every secret вҖ” assume it leaked. **Prevention:** if `777` is your fix, you haven't found the bug. `namei -l` finds the bug.

---

### Mistake 4 вҖ” Setting a setuid bit on a script

**Wrong:** `sudo chmod 4755 /opt/deploy/restart-api.sh` ("now anyone can restart the API as root").
**Right:** a pinned `sudoers` rule (edit with `sudo visudo -f`, never a plain editor):
```
deploy ALL=(root) NOPASSWD: /usr/bin/systemctl restart myapi
```
**Root cause:** the kernel **ignores** setuid/setgid on any file executed via a shebang interpreter, so the script just runs as you вҖ” the "fix" silently does nothing (the *good* outcome). The *bad* outcome: someone routes around it with a setuid **C wrapper** calling `system("systemctl restart myapi")`. `system()` runs `/bin/sh -c`, `sh` respects `$PATH`, and an attacker who prepends `~/evil` to `$PATH` gets a root shell вҖ” textbook privesc.
**Diagnose:** `find / -xdev -perm -4000 -type f 2>/dev/null` вҖ” anything not from a distro package is an incident. **Prevention:** `sudo` with an absolute-path, argument-pinned rule (topic 32).

---

### Mistake 5 вҖ” Assuming the owner always has the most access

**Wrong:** "I own the file, so I can obviously write it."
**Right:** the kernel picks **exactly one** class and checks **only that one**. Match `user` and the group/other bits are never consulted, even if they'd grant more.

```bash
touch f && chmod 0466 f    # -r--rw-rw-
echo hi > f                # bash: f: Permission denied  вҶҗ YOU own it, group+other have w
```

**Root cause:** `generic_permission()` does `if (uid == i_uid) { mode >>= 6; }` and *returns* on that branch вҖ” no fall-through to group. **Fix:** `chmod u+w f`. In production this only bites with a genuinely inverted mode, which is exactly what a careless `chmod o+w` on a file you own produces.

---

### Mistake 6 вҖ” Docker bind mounts don't care about your usernames

**Wrong:** `ls -l` shows `node:node` in the container and `1000:1000` on the host, so you assume they're the same identity.
**Right:** the inode stores only the **numeric** UID/GID (topic 04); names are a userspace `/etc/passwd` lookup, and the container has a *different* `/etc/passwd`.

```bash
ls -ln /var/www/app/uploads    # -n shows NUMBERS вҖ” the honest view
# drwxrwxr-x 2 1001 33 ... uploads    (host: deploy:www-data)
# Inside `docker run -v ...:/app/uploads node:18`, node is uid 1000 вү  1001,
# and gid 33 is 'tape' in Debian, not www-data вҶ’ EACCES with a misleading `ls -l`.
```
**Fix:** align the numbers вҖ” `docker run --user 1001:33 ...`. Always debug mounts with `ls -ln`, never `ls -l`.

## Hands-On Proof

```bash
# PROVE IT: the mode is 12 bits in the inode. %f is the RAW i_mode in hex.
touch /tmp/p && chmod 4755 /tmp/p && stat -c 'octal=%a full=%f symbolic=%A' /tmp/p
# octal=4755 full=8fed symbolic=-rwsr-xr-x
#            ^^^^ 0x8fed = 1000(regular file) 100(setuid) 111 101 101(rwxr-xr-x)

# PROVE IT: umask math, not magic.
umask 022; touch /tmp/a; mkdir /tmp/b; stat -c '%a %n' /tmp/a /tmp/b
# 644 /tmp/a   вҶҗ 0666 & ~022      755 /tmp/b   вҶҗ 0777 & ~022
umask 077; touch /tmp/c; mkdir /tmp/d; stat -c '%a %n' /tmp/c /tmp/d
# 600 /tmp/c   вҶҗ 0666 & ~077      700 /tmp/d   вҶҗ 0777 & ~077
umask 022

# PROVE IT: programs request 0666 for FILES вҖ” that's why x is never set by default.
strace -e trace=openat,mkdir bash -c 'echo hi >/tmp/e; mkdir /tmp/f' 2>&1 | grep -E 'tmp/(e|f)'
# openat(AT_FDCWD, "/tmp/e", O_WRONLY|O_CREAT|O_TRUNC, 0666) = 3   вҶҗ 0666, NOT 0777. Ever.
# mkdir("/tmp/f", 0777) = 0                                        вҶҗ dirs DO ask for 0777

# PROVE IT: EACCES comes from the KERNEL, errno 13 вҖ” Node just relays it.
chmod 000 /tmp/e
strace -e trace=openat cat /tmp/e 2>&1 | grep 'tmp/e'
# openat(AT_FDCWD, "/tmp/e", O_RDONLY) = -1 EACCES (Permission denied)   вҶҗ the KERNEL said no
node -e 'try{require("fs").readFileSync("/tmp/e")}catch(e){console.log(e.code,e.errno)}'
# EACCES -13     вҶҗ the same kernel refusal in a JavaScript costume

# PROVE IT: the kernel checks ONE class, not the union.
touch /tmp/g && chmod 0466 /tmp/g       # -r--rw-rw-
echo x > /tmp/g
# bash: /tmp/g: Permission denied        вҶҗ you OWN it, group+other have w, still denied

# PROVE IT: x on a dir is TRAVERSE; r is LIST вҖ” independent.
mkdir -p /tmp/t/sub && echo secret > /tmp/t/sub/s.txt && chmod 111 /tmp/t
ls /tmp/t              # cannot open directory: Permission denied   (no r)
cat /tmp/t/sub/s.txt  # secret                                      (x let you traverse)
chmod 755 /tmp/t

# PROVE IT: /tmp is 1777, and setuid is REAL on binaries but IGNORED on scripts.
stat -c '%a %A %n' /tmp                        # 1777 drwxrwxrwt /tmp
ls -l /usr/bin/passwd                          # -rwsr-xr-x  вҶҗ real setuid binary
printf '#!/bin/bash\nid -u\n' | sudo tee /tmp/su.sh >/dev/null
sudo chown root /tmp/su.sh && sudo chmod 4755 /tmp/su.sh
/tmp/su.sh                                     # 1001  вҶҗ setuid IGNORED on scripts. Still you.

# PROVE IT: setgid on a dir changes the group of NEW files; namei finds a guilty component.
sudo mkdir /tmp/sg && sudo chgrp adm /tmp/sg && sudo chmod 2777 /tmp/sg
touch /tmp/sg/x && stat -c '%U:%G %n' /tmp/sg/x     # deploy:adm  вҶҗ group INHERITED from dir
sudo mkdir -p /tmp/a/b/c && sudo chmod 700 /tmp/a/b && sudo touch /tmp/a/b/c/f
namei -l /tmp/a/b/c/f | grep -E 'b$|^ *d.* a$'
#  drwx------ root root b   в—Җ no x for you вҶ’ EACCES on anything below, regardless of f's mode
```

**macOS note (you're on macOS locally):** `chmod`, `chown`, `umask`, and octal notation all work identically. But: `stat -c` is GNU вҖ” on macOS use `stat -f '%Lp %Su:%Sg %N'`. `namei` **does not exist** (install `util-linux` via brew, or just use a Linux container). `ls -l` may show a trailing `@` (extended attributes) or `+` (an ACL) after the mode вҖ” those are macOS-specific and do not exist on a stock Ubuntu box. `chattr`/`lsattr` are Linux-only.

**Alpine/BusyBox note:** BusyBox `chmod` supports symbolic and octal, but **not** the capital-`X` mode. Use the two-pass `find -type d/-type f` form instead. `namei` needs `apk add util-linux`. Many Alpine images run as root by default, which is precisely why permission bugs stay hidden until you correctly add a `USER node` line to your Dockerfile.

## Quick Reference Card

| Command | What it does | Key flags |
|---|---|---|
| `ls -l` | Mode, links, owner, group, size, name | `-d` on the **dir itself**, not its contents; `-n` show **numeric** uid/gid (essential for Docker) |
| `ls -ld /tmp` | Inspect a directory's own mode | `drwxrwxrwt` = 1777 |
| `stat -c '%a %A %U:%G %n' F` | Mode octal + symbolic + owner | `%a` octal, `%A` symbolic, `%f` raw hex i_mode, `%U`/`%G` names |
| `chmod 644 F` | Set mode, octal | 4-digit form (`4755`) sets special bits; **3-digit form CLEARS them** |
| `chmod u+x,go-w F` | Set mode, symbolic | `u`/`g`/`o`/`a`; `+`/`-`/`=`; `s` setuid/setgid, `t` sticky |
| `chmod -R u=rwX,g=rX,o=rX D` | Fix a whole tree correctly | **capital `X`** = x on dirs (and already-executable files) only |
| `chmod 2775 D` | setgid dir вҖ” new entries inherit its group | the deploy-directory bit |
| `chmod 1777 D` | sticky dir вҖ” only owners delete their own | `/tmp`, upload spools |
| `chown u:g F` | Change owner and group | `-R` recursive; **root only**; `-h` affect the symlink itself |
| `chgrp g F` | Change group only | usable **without root** if you own F and are in group g |
| `umask` | Show/set the mask on new files | `-S` symbolic; files = `0666 & ~umask`, dirs = `0777 & ~umask` |
| `namei -l PATH` | **Mode + owner of EVERY path component** | в—Җ the `EACCES` debugger. `-m` also shows the mode string. |
| `find -perm -4000` | Every setuid file | `-4000` setuid, `-2000` setgid, `-1000` sticky |
| `find -perm /o+w` | Anything world-writable | `/` = ANY of these bits; `-` = ALL of these bits; bare = EXACT |
| `sudo -u USER test -w F` | Ask the kernel directly: can this user write F? | `-r` read, `-w` write, `-x` exec/traverse |
| `getfacl` / `setfacl` | Per-user ACLs beyond the 9 bits | a `+` at the end of `ls -l`'s mode means an ACL exists |
| `chattr +i F` | Immutable вҖ” even **root** cannot unlink it | `lsattr` to view; ext2/3/4 only |

## When Would I Use This at Work?

### Scenario 1: The deploy that broke SSH
A teammate hits `EACCES` on a deploy and "fixes" it with `sudo chmod -R 777 /home/deploy`. The deploy works. The next SSH login fails with `Permission denied (publickey)` and nobody can get in. **Cause:** `sshd` runs with `StrictModes yes` and **refuses** to read `~/.ssh/authorized_keys` when the home directory or the key file is group/world-writable вҖ” it assumes another user could have injected a key. Fix (from the cloud console's serial terminal, since SSH is gone): `chmod 755 /home/deploy && chmod 700 /home/deploy/.ssh && chmod 600 /home/deploy/.ssh/authorized_keys`. Then rotate the keys, because for the window it was 777, anyone on the box could have appended their own.

### Scenario 2: nginx returns 403 on your static assets
Your Node build drops files into `/var/www/app/public`. nginx (running as `www-data`) returns `403 Forbidden`, and `/var/log/nginx/error.log` says `Permission denied ... open() "/var/www/app/public/main.js" failed (13: Permission denied)`. `namei -l /var/www/app/public/main.js` shows `drwx------ deploy deploy app` вҖ” the deploy user's umask was `077`, so the build directory came out `700`. Fix: set `UMask=0022` in the deploy unit (or `umask 022` in the deploy script), `chmod 2755` the tree so the group sticks, and add `www-data` to the `deploy` group. Errno 13 in the nginx log is the *same* errno 13 Node reports вҖ” same kernel check, different messenger.

### Scenario 3: The audit finds a setuid binary you didn't put there
`find / -xdev -perm -4000 -type f` returns `/tmp/.cache/x` owned `root:root`, mode `-rwsr-xr-x`. That is not a distro file. Someone got a shell, and left themselves a root backdoor: any user who runs it becomes root. Rebuild the box вҖ” but first, note that the file's **ctime** (topic 04, the timestamp that `touch` cannot forge) tells you exactly when the setuid bit was applied, which anchors the whole incident timeline. Prevention: mount `/tmp` and any user-writable volume with `nosuid,nodev,noexec` in `/etc/fstab`, which makes the kernel ignore setuid bits on that entire filesystem.
